# Workflow агентов: разработка ML-урока

Переиспользуемый pipeline для пар уроков вида `Урок_21_..._Теория` / `Урок_22_..._Практика`: теория + упражнения + практика на одной сквозной модели и датасете.

## Артефакты на выходе

| Файл | Назначение | Кто пишет код |
|------|------------|---------------|
| `Урок_N_...ipynb` | Занятие **N**: концепции, сквозной пример, демо-код | преподаватель/агент |
| `Тест_к_теории.ipynb` / `Тесты_к_теории.ipynb` | Занятие **N**: короткие проверки теории | студент |
| `Урок_{N+1}_..._Практика.ipynb` | Следующий урок (~90 мин): полный пайплайн | авторский вариант с ответами и критериями |
| `*_hand_solution.ipynb` | Локальная пустая копия для выдачи студентам | студент |

**Не коммитить** `*_hand_solution.ipynb`; они игнорируются через `.gitignore`.

## Ограничения (из реального проекта)

- **Не редактировать** `plan.md`, если пользователь явно не просит.
- **Не коммитить** без запроса пользователя.
- **Ручные правки в ноутбуке** — точечные изменения по `# TODO` в ячейках; **не перегенерировать** файл целиком, если пользователь уже редактировал материал.
- **Одна главная модель** на сквозной пример (kNN, логрег, …); baseline (`DummyClassifier`) — отдельный сюжет, не замена модели.
- **Нумерация разделов** — только последовательная (1, 2, 3, …; без «2а»).
- **Нумерация занятий:** теория и упражнения — **N**; практика — **всегда следующий урок**. В этом разделе курса пары идут с занятия **21**: 21/22, 23/24, …, 35/36. В заголовках, metadata `name` и перекрёстных ссылках соблюдать это правило.
- **Имена папок:** использовать формат `Урок_N_<тема>_Теория` и `Урок_{N+1}_<тема>_Практика`, как в экспортной папке DS-course.

## Нумерация занятий по папке

Номер в имени папки `Урок_N_...` задаёт номер **теории** или **практики**:

| Тема | Теория **N** | Практика |
|------|--------------|----------|
| ML workflow и kNN | 21 | 22 |
| Feature Engineering | 23 | 24 |
| Линейная регрессия | 25 | 26 |
| Логистическая регрессия | 27 | 28 |
| Переобучение и валидация | 29 | 30 |
| Решающее дерево | 31 | 32 |
| Bagging / Random Forest | 33 | 34 |
| Boosting | 35 | 36 |

**Правило:** практика = `N + 1`. Упражнения — тот же **N**, что и теория.

При создании материалов сверять заголовки ноутбуков и `plan.md` с этой формулой.
- **Термины** вводятся до первого использования; без дублирования одного и того же блока в соседних ячейках.
- **Практика**: основной блокнот хранит условия, авторские ответы и подробные критерии LLM-проверки; пустые копии для выдачи студентам называются `*_hand_solution.ipynb` и не коммитятся.
- **API sklearn**: актуальные параметры (напр. `l1_ratio` вместо deprecated `penalty`).

## Ссылки на разделы теории

В текстах уроков (теория, упражнения, практика) **не использовать** знак §.

| Случай | Формат | Пример |
|--------|--------|--------|
| один раздел | **п.** | «по мотивам п. 4 теории», «см. п. 7» |
| несколько разделов | **пп.** | «пп. 4 и 7», «пп. 3–5», «пп. 9–11» |

Заголовки разделов в теории остаются с арабской цифрой: `## 7. Метрика…`. Ссылка в соседних ноутбуках — через **п.** / **пп.**, не через §.

### Минимум перекрёстных ссылок (читаемость)

Ссылки **п.** / **пп.** нужны для навигации, но **не должны засорять** текст. Читатель идёт по ноутбуку последовательно — не дублировать номера соседних разделов.

| Где | Правило |
|-----|--------|
| **Теория** | Не ссылаться на разделы **N±1…3** («см. п. 5» сразу перед п. 5). Вместо этого: «ниже», «в следующем разделе», «далее». Ссылку **п.** оставлять только при **скачке** на 4+ раздела или на другой ноутбук. |
| **Упражнения** | **Не** писать `(п. N)` в заголовке **каждой** задачи. Одна строка в шапке: «по теории занятия N». Номер раздела — только если без него непонятно, о какой теме задача (редко). |
| **Практика** | В шапке — ссылка на ноутбук теории (`…_theory.ipynb`, занятие N). **Не** повторять «По п. 12» в каждом задании: шаги и критерий должны быть самодостаточны. |
| **Чек-лист в конце теории** | Допустимы **п.** / **пп.** — это оглавление-напоминание, не сплошной текст. |

**Плохо:** «Данные в п. 8; сравним в п. 12; OOB — п. 14 (подробнее п. 24)».  
**Хорошо:** «Данные и split — в следующем блоке»; в практике: «См. `Урок_33_Ансамбли_Bagging_Случайный_лес.ipynb`».

## Один концепт на раздел теории

- Каждый H2 раскрывает **один** крупный концепт (workflow, metrics, train-split, cv, leakage, features, missing, categorical, outliers, multicollinearity, scaling, imbalance, models, regularization, hyperparams, overfitting, importance).
- Не объединять в одном H2 независимые темы («Baseline и дисбаланс», «Масштаб и регуляризация»). Исключение — чек-листы и roadmap.
- После split — перенумеровать H2 и обновить **п.** / **пп.** во всей папке занятия.
- Чек-лист: `.cursor/rules/theory-single-concept-sections.mdc`.
- **Статичные таблицы** — markdown, не code: `.cursor/rules/theory-static-tables.mdc`.

## Canvas карты концептов

`canvases/ml-theory-concepts-map.canvas.tsx` — карта «крупный концепт × занятие»; каждая плашка = раздел `## N.` теории.

После **создания или изменения оглавления** теории (новые разделы, переименование, split, перенумерация):

1. Обновить массив `LESSONS` для занятия **N**: плашки = актуальные заголовки разделов.
2. Для переименованных плашек добавить записи в `LABEL_RENAME` / `LABEL_SPLIT` (миграция сохранённой раскладки пользователя).
3. Новому сквозному концепту (если появился) — строка в `BIG_CONCEPTS` + правило в `assignBigConcept`.
4. При изменении миграций поднять версию ключа `useCanvasState` (напр. `label-split-v2` → `v3`), иначе миграция не применится к сохранённому состоянию.

## Pipeline агентов

```
[0. Brief] → [1. Architect] → [2. Theory] → [3. Exercises] ─┐
                              → [4. Practice] ──────────────┤
                              → [4b. Solution] ─────────────┤
                              → [5. Tech Review] ───────────┤→ [7. Final Editor]
                              → [6. Methodology Review] ────┤
                              → [6b. Philologist Review] ───┤
                              → [6c. Student Reader] ───────┘
```

Агенты 3, 4 и **4b** можно запускать параллельно после Theory. Review-фазы идут после сборки черновиков; филолог проверяет язык; **школьник-читатель** — понятность; финальный редактор собирает всё вместе.

## Агент 0 — Brief (вход пользователя)

**Цель:** зафиксировать постановку до кода.

**Собрать:**
- номер занятия теории **N** (практика = **следующий урок**, упражнения = **N**); для папки `{NN}_…` обычно **N = NN × 2 − 1**;
- тема и **главная модель** сквозного примера;
- датасет (встроенный sklearn / свой);
- длительность практики;
- что уже есть в курсе (предпосылки);
- явные запреты (не трогать `plan.md`, формат практики и т.д.).

**Выход:** `LESSON_BRIEF.md` (1 страница) — см. шаблон в [brief-template.md](brief-template.md).

---

## Агент 1 — Architect (архитектор урока)

**Роль:** методист + ML-эксперт. План структуры, не тексты.

**Вход:** Brief + существующие материалы темы + соседние уроки курса.

**Задачи:**
1. Составить **чек-лист покрытия** темы (как для workflow: типы задач, train/val/test, baseline, FE, preprocessing, метрики, error analysis, test).
2. Выбрать **сквозной датасет** и обосновать (масштабы признаков, бинарная классификация, интерпретируемость).
3. Разбить теорию на **п. 1–N** (15–25 разделов): порядок введения терминов важнее «красоты» кода. **Train/validation/test** — сразу после загрузки сквозного датасета (обычно п. 4), до постановки и baseline.
4. Спланировать практику: **10–12 заданий** с таймингом (~90 мин).
5. Спланировать упражнения: **10–15** коротких задач, последние 2–3 — по сквозной модели.

**Выход:** `LESSON_OUTLINE.md` — оглавление разделов, таблица заданий практики, список упражнений.

**Критерий готовности:** нет термина в п. 2, который впервые определяется только в п. 9; порядок этапов пайплайна одинаков в теории, упражнении 11 и практике; все примеры и таблицы сценариев привязаны к одному сквозному датасету.

## Агент 2 — Theory Author (теория)

**Роль:** автор `{topic}_theory.ipynb`.

**Вход:** Outline + Brief.

**Правила оформления:**
- Заголовок: `Занятие N. {Тема} на примере {Модель}` (N — номер теории из brief).
- п. 3 (или раньше): модель + признак/метка + сквозной датасет **одним блоком** (без дублирующей ячейки «сквозной датасет»).
- После загрузки датасета — **короткий код** евклидова расстояния по 2–3 признакам (если модель опирается на расстояние, напр. kNN).
- **Сквозной пример** — все формулировки, таблицы сценариев и демо-код про **один датасет**; не смешивать «клиентов», «квартиры» и медицинский кейс в одном уроке.
- Метрики: recall/precision/accuracy **до** baseline; для медицинских задач — `pos_label` и FN/FP **до** confusion matrix; явно объяснить, что **`pos_label`** задаёт «положительный» класс для recall.
- Train/validation/test — **сразу после** загрузки датасета (п. 4), **до** постановки задачи и baseline; в тексте назвать `train_test_split` из `sklearn.model_selection`.
- Раздел «offline-метрика vs прод» — **не включать**, если тема не про A/B и мониторинга в проде достаточно в п. «внедрение».
- `fit` / `transform` / `StandardScaler` — **до** масштабирования в коде.
- Инжиниринг признаков: явно — сравнение идей только на **validation**, test не для выбора признаков.
- Анализ ошибок: **markdown-таблица** confusion matrix + код с подписанным `DataFrame`; по группам — **доля ошибок внутри группы**, с пометкой, что доли по группам не суммируются в 1.
- Чек-лист в конце — краткое напоминание по разделам (допустимы **п.** / **пп.**); в основном тексте — минимум ссылок (см. «Минимум перекрёстных ссылок»).
- **Один концепт на H2** — не смешивать baseline и imbalance, scaling и regularization и т.п. в одном разделе (см. `.cursor/rules/theory-single-concept-sections.mdc`).
- Демо-код: один `random_state`, перезапускаемые outputs.
- Убрать дубли: таблица markdown **или** DataFrame-код, не оба с одинаковым содержимым.
- Комментарии `# TODO` от автора/рецензента — **удалить** после исправления, не оставлять в финальной версии.

**Выход:** `{topic}_theory.ipynb` + выполненные ячейки.

## Агент 3 — Exercises Author (упражнения)

**Роль:** автор `{topic}_exercises.ipynb`.

**Вход:** Outline + готовая (или черновая) теория.

**Правила:**
- Импорты — **в начале** ноутбука.
- Формат: `variable = ...` + `assert` + `print('Верно')`.
- Каждое упражнение — **одна идея** из теории; в шапке ноутбука — ссылка на `{topic}_theory.ipynb`, не `(п. N)` в каждом заголовке.
- Задания по модели (k, scaling, `pos_label`) — в конце; номер раздела только если тема неочевидна из формулировки.
- Ссылка на практику — **занятие следующий урок** (не N).
- Не дублировать практику: упражнения проверяют понимание, не полный пайплайн.

**Выход:** `{topic}_exercises.ipynb`.

---

## Агент 4 — Practice Author (практика)

**Роль:** автор `{topic}_practice.ipynb`.

**Вход:** Outline + теория (для ссылок п. / пп.).

**Правила:**
- Заголовок: **Занятие следующий урок** (номер теории + 1), модель и время.
- Ссылки на теорию — **занятие N**; на практику в упражнениях — **занятие следующий урок**.
- Ячейка **«Дано»** — только данные/константы; **не менять**.
- Задания 0–9: markdown + **пустая** code-ячейка.
- Каждое задание: цель, шаги, **критерий**; ссылка на теорию — **один раз** в шапке практики, не в каждом задании (см. «Минимум перекрёстных ссылок»).
- Цикл практики = цикл п. 2 теории (без лишнего EDA/split на английском без расшифровки).
- `RANDOM_STATE` и размеры split **согласованы** с теорией.
- Итог — короткий, без дублирования таблицы из теории.

**Выход:** `{topic}_practice.ipynb`.

---

## Агент 4b — Solution Author (решение практики)

**Роль:** автор `{topic}_practice_solution.ipynb` **только для преподавателя**.

**Вход:** `{topic}_practice.ipynb` + `{topic}_theory.ipynb`.

**Правила:**
- Те же задания 0–9, но **заполненный код** вместо пустых ячеек.
- В шапке: «Только для преподавателя. Не выдавать студентам».
- Эталонные значения: split, baseline, `best_k`, recall с правильным `pos_label`.
- Для confusion matrix явно указать положительный класс (TP/FN/FP/TN).
- Перезапустить ноутбук; outputs актуальны.

**Выход:** `{topic}_practice_solution.ipynb`.

## Агент 5 — Technical Reviewer (технический рецензент)

**Роль:** ML-инженер. Код должен работать и быть честным.

**Чек-лист:**
- [ ] Все code-ячейки теории выполняются без ошибок.
- [ ] Outputs актуальны (нет stale после смены датасета/модели).
- [ ] Split sizes, baseline, метрики — согласованы между theory/practice.
- [ ] Гиперпараметры не захардкожены, если в тексте «выбрали по validation».
- [ ] `recall_score(..., pos_label=...)` для нужного класса.
- [ ] Нет утечки: scaler/kNN fit только на train.
- [ ] sklearn API актуален.
- [ ] JSON ноутбуков валиден.
- [ ] `{topic}_practice_solution.ipynb` выполняется и совпадает с критериями практики.

**Выход:** список блокеров / исправления в файлах.

---

## Агент 6 — Methodology Reviewer (методист)

**Роль:** школьный/вузовский методист ML.

**Чек-лист:**
- [ ] Каждый термин определён **раньше** первого использования.
- [ ] Нет дублирования: датасет, цикл workflow, таблицы типов задач, итог практики.
- [ ] «Validation» как выборка vs «сравнение гипотез» — не путаются.
- [ ] Confusion matrix: явно указан положительный класс (для медицины — часто класс 0); markdown-таблица + подписанный DataFrame в коде.
- [ ] Примеры и таблицы сценариев — только про сквозной датасет (без посторонних «квартир»/«клиентов»).
- [ ] Train/val/test введены до baseline и до постановки, если split показан в коде сразу после загрузки данных.
- [ ] Практика не требует кода, который ещё не введён в теории.
- [ ] Задания практики укладываются в заявленное время.
- [ ] Названия занятий отражают модель/тему.
- [ ] Практика — **следующий урок**, теория и упражнения — **N**; ссылки между ноутбуками согласованы.
- [ ] Перекрёстные ссылки: **п.** / **пп.**, не §.
- [ ] Каждый H2 теории — один концепт; нет смешанных заголовков («X и Y») кроме чек-листов/roadmap.
- [ ] После изменения оглавления теории ссылки в exercises/practice/solution совпадают с номерами разделов.

**Выход:** список правок + подтверждение готовности.

## Агент 6b — Philologist Reviewer (филолог)

**Роль:** редактор русского языка и учебного стиля. Не меняет ML-смысл без согласования с техническим/методическим review.

**Чек-лист:**
- [ ] Русский язык: орфография, пунктуация, согласование, падежи, единообразие кавычек и тире.
- [ ] Стиль: ясные короткие фразы, без канцелярита и лишней англификации.
- [ ] Термины: один термин — одно написание (`baseline`, `train`, `validation`, `test`, `kNN`); если есть русский эквивалент, он введён рядом.
- [ ] Тон: дружелюбный, учебный, без снисходительности и «магических» формулировок.
- [ ] Заголовки и критерии заданий сформулированы как действия студента.
- [ ] Markdown аккуратен: списки параллельны, нет лишних повторов, нет длинных перегруженных абзацев.
- [ ] Ссылки на разделы теории: **п.** / **пп.**, не §.
- [ ] Не упрощать так, чтобы терялся технический смысл; сомнительные ML-формулировки помечать вопросом, а не переписывать произвольно.

**Выход:** языковые правки в ноутбуках + краткий список спорных мест, если они остались.

## Агент 6c — Student Reader (школьник-читатель)

**Роль:** читатель целевой аудитории (школьник / начинающий студент). Не ML-эксперт. Цель — поймать места, где текст **понимается не с первого раза**, и **исправить формулировки**, чтобы типичные вопросы не возникали.

**Вход:** `{topic}_theory.ipynb` (и при необходимости practice/exercises — только ссылки и формулировки заданий).

**Процесс:**
1. Прочитать теорию **последовательно**, как на занятии.
2. Записать **вопросы школьника** по каждому непонятному месту («а зачем три части?», «что такое pos_label?»).
3. Для каждого вопроса — **точечная правка** markdown (не перегенерировать ноутбук): определение, аналогия, мини-пример с числами, таблица «термин → смысл».
4. Сдать **отчёт**: таблица «вопрос → что было неясно → как исправлено».

**Чек-лист понятности:**
- [ ] Английские термины (train, baseline, workflow, sklearn) при **первом** упоминании — с русским пояснением в скобках или таблицей.
- [ ] **Train / validation / test**: зачем **три** части, а не две; простая аналогия (учебник / пробник / контрольная).
- [ ] Термины **метка**, **признак**, **класс 0/1** — до кода и метрик; для «класс 0 = злокач.» — явно, что это соглашение датасета, а не «0 = плохо».
- [ ] **`pos_label`**: «positive» ≠ «хороший класс»; это «про какой класс считаем recall».
- [ ] **Recall / precision / FN / FP** — с числовым микро-примером и таблицей ошибок **до** confusion matrix в коде.
- [ ] **`fit` / `transform`**, **`StandardScaler`** — что делает по шагам, почему `fit` только на train.
- [ ] **Confusion matrix** — как читать: строки = истина, столбцы = прогноз.
- [ ] Нет **опережающих** ссылок без пояснения (если термин вводится позже — однострочное пояснение или «ниже», без лишних «п. N» на соседние разделы).
- [ ] Примеры и аналогии **из одного** сквозного датасета; чужие домены (клиенты, возраст) — только если явно помечены как «другой пример».
- [ ] **Baseline** — зачем сравнивать с «всегда самый частый класс»; intuition до кода.

**Выход:** правки в `{topic}_theory.ipynb` + **STUDENT_REVIEW.md** (вопросы и исправления). Код менять только если текст ссылается на непонятный output.

**Не делать:** упрощать до потери смысла; удалять термины, нужные для практики.
