# Методика преподавания One-Hot Encoding: от dummy-переменных до Pipeline

## Назначение методики

Методика описывает последовательность изучения one-hot encoding на одном сквозном примере. Ученик проходит путь от ручного создания dummy-переменных до готовой цепочки `ColumnTransformer` + `Pipeline`.

Главная цель занятия - не запомнить несколько команд, а понять общий принцип работы с обучаемыми преобразованиями:

```text
на train изучаем правило преобразования
                 ↓
на новых данных применяем то же правило
                 ↓
модель всегда получает одинаковый набор признаков
```

После занятия ученик должен уметь:

1. Отличать номинальный категориальный признак от числового и порядкового.
2. Объяснять, какую информацию добавляет произвольная замена категорий числами.
3. Строить dummy-признаки вручную и с помощью `pd.get_dummies`.
4. Объяснять различие между `fit`, `transform` и `fit_transform`.
5. Применять `OneHotEncoder` к новым и неизвестным категориям.
6. Выбирать нужные столбцы с помощью `ColumnTransformer`.
7. Связывать подготовку признаков и модель в `Pipeline`.

## Методический замысел

Тема строится как последовательное решение одной задачи. Каждый новый инструмент появляется только после того, как возможностей предыдущего становится недостаточно:

| Этап | Что уже умеем | Какое ограничение обнаруживаем | Следующий инструмент |
|---|---|---|---|
| Категории | Видим текстовые значения | Большинство изучаемых моделей ожидает числа | Числовое представление |
| Произвольные коды | Можем заменить слова числами | Коды задают модели искусственный порядок и расстояния | Dummy-переменные |
| Dummy вручную | Понимаем смысл бинарных столбцов | Ручной код неудобен для нескольких категорий | `pd.get_dummies` |
| `pd.get_dummies` | Быстро кодируем одну таблицу | При раздельной обработке набор столбцов зависит от данных | `OneHotEncoder` |
| `OneHotEncoder` | Запоминаем категории на train | В таблице есть признаки разных типов | `ColumnTransformer` |
| `ColumnTransformer` | Готовим всю таблицу | Преобразование и модель пока запускаются отдельно | `Pipeline` |

Такой порядок сохраняет причинную связь между понятиями: новый объект вводится не как еще одна команда, а как ответ на уже понятную проблему.

## Сквозной учебный пример

На протяжении занятия используется таблица с четырьмя признаками:

```python
import pandas as pd

students = pd.DataFrame({
    "city": ["Москва", "Казань", "Омск", "Москва"],
    "school_type": ["лицей", "гимназия", "обычная", "лицей"],
    "age": [15, 16, 17, 16],
    "score": [78, 85, 64, 91]
})
```

Признаки выполняют разные роли:

| Признак | Тип | Планируемая обработка |
|---|---|---|
| `city` | номинальный категориальный | one-hot encoding |
| `school_type` | номинальный категориальный | one-hot encoding |
| `age` | числовой | оставить без изменений |
| `score` | числовой | оставить без изменений |

В рамках этого занятия не добавляется масштабирование. Оно решает другую задачу и не требуется для понимания механики one-hot encoding, `ColumnTransformer` и `Pipeline`.

## Этап 1. Уточнить смысл категориального признака

### Учебная задача

Ученик должен увидеть, что значения `city` обозначают группы, а не величины.

Преподаватель предлагает сравнить признаки:

- `age`: значения можно упорядочивать, вычитать и сравнивать по разности;
- `city`: значения различаются, но сами названия городов не задают числовую шкалу.

Затем обсуждается наивная кодировка:

```text
Москва -> 0
Омск   -> 1
Казань -> 2
```

Проблема этой записи формулируется точно: модель получает не только обозначения категорий, но и дополнительную структуру:

- `Москва < Омск < Казань`;
- расстояние от Москвы до Казани равно 2;
- расстояние от Омска до Казани равно 1.

Эта структура возникла из способа нумерации, а не из исходных данных.

### Диагностический вопрос

> Если поменять местами числовые коды Москвы и Омска, изменится ли смысл исходного признака `city`?

Ожидаемый ответ: нет, города останутся теми же. Однако числовой столбец изменит порядок и расстояния между значениями, поэтому некоторые модели могут построить другой результат. Значит, произвольные целые коды не являются нейтральным представлением номинального признака.

Здесь важно отдельно проговорить: one-hot encoding рассматривается именно для номинальных категорий. У порядковых категорий, например `низкий`, `средний`, `высокий`, может быть другое представление, если порядок действительно важен для задачи.

## Этап 2. Построить dummy-переменные вручную

### Учебная задача

Ученик должен понять смысл преобразования до знакомства с готовой функцией.

Для одного столбца `city` создаются три бинарных вопроса:

```python
city_dummy = pd.DataFrame({
    "city_Москва": (students["city"] == "Москва").astype(int),
    "city_Казань": (students["city"] == "Казань").astype(int),
    "city_Омск": (students["city"] == "Омск").astype(int)
})

city_dummy
```

Каждый столбец отвечает на один вопрос: относится ли объект к этой категории?

| city_Москва | city_Казань | city_Омск |
|---:|---:|---:|
| 1 | 0 | 0 |
| 0 | 1 | 0 |
| 0 | 0 | 1 |
| 1 | 0 | 0 |

### Вопросы для обсуждения

1. Что означает единица в столбце `city_Казань`?
2. Почему для одной исходной строки единица находится только в одном из трех столбцов?
3. Возникает ли теперь порядок между Москвой, Казанью и Омском?
4. Сколько новых столбцов потребуется для десяти разных городов?

Последний вопрос подводит к автоматизации: идея проста, но создавать каждый столбец вручную неудобно.

## Этап 3. Автоматизировать преобразование через `pd.get_dummies`

### Учебная задача

Связать понятую вручную операцию с готовым инструментом `pandas`.

```python
students_dummies = pd.get_dummies(
    students,
    columns=["city", "school_type"],
    dtype=int
)

students_dummies
```

Ученики сравнивают результат с ручным кодированием и находят:

- исходный категориальный столбец исчез;
- для каждой найденной категории появился бинарный столбец;
- числовые столбцы `age` и `score` сохранились;
- названия новых столбцов содержат исходный признак и категорию.

На этом этапе `pd.get_dummies` представляется как удобный способ увидеть и проверить результат преобразования. Не нужно сразу переходить к ограничениям: сначала ученики должны уверенно прочитать полученную таблицу.

## Этап 4. Обнаружить ограничение `pd.get_dummies`

### Учебная задача

Показать, почему в машинном обучении важно не только преобразовать таблицу, но и сохранить правило преобразования.

```python
train = pd.DataFrame({
    "city": ["Москва", "Казань", "Омск"]
})

new_data = pd.DataFrame({
    "city": ["Москва", "Тула"]
})

train_dummies = pd.get_dummies(train, dtype=int)
new_data_dummies = pd.get_dummies(new_data, dtype=int)
```

После выполнения ученики сравнивают `train_dummies.columns` и `new_data_dummies.columns`.

В обучающей таблице есть признаки:

```text
city_Казань, city_Москва, city_Омск
```

В новой таблице:

```text
city_Москва, city_Тула
```

Вывод формулируется на уровне требования модели:

> При обучении и при прогнозе модель должна получать признаки с одинаковыми названиями и одинаковым смыслом.

`pd.get_dummies` строит результат по категориям той таблицы, которую обрабатывает в данный момент. Функция не хранит внутри себя список категорий, найденных на train. В учебных и исследовательских задачах это полезный инструмент, но для полноценной ML-цепочки удобнее преобразователь с состоянием.

## Этап 5. Ввести `OneHotEncoder` через `fit` и `transform`

### Учебная задача

Показать, что преобразователь сначала изучает категории на train, а затем использует сохраненное правило.

```python
from sklearn.preprocessing import OneHotEncoder

encoder = OneHotEncoder(
    handle_unknown="ignore",
    sparse_output=False
)

encoder.fit(train)
encoder.categories_
```

После `fit` преподаватель показывает `encoder.categories_` и задает вопрос:

> Какие данные сохранил энкодер?

Ожидаемый ответ: список категорий каждого обучаемого столбца.

Затем одно правило применяется к обеим таблицам:

```python
train_encoded = encoder.transform(train)
new_data_encoded = encoder.transform(new_data)
```

Ключевое различие методов:

| Метод | Что делает | Где используется |
|---|---|---|
| `fit(X_train)` | запоминает категории из train | только при обучении |
| `transform(X)` | кодирует по уже сохраненным категориям | для train, validation, test и новых данных |
| `fit_transform(X_train)` | выполняет `fit`, затем `transform` | сокращенная запись для train |

Нельзя выполнять новый `fit` на validation, test или поступивших после обучения данных. Иначе правило подготовки признаков изменится.

### Неизвестная категория

В `new_data` есть Тула, которой не было на этапе `fit`. Параметр

```python
handle_unknown="ignore"
```

позволяет применить сохраненную схему без ошибки. Для строки с Тулой все столбцы, соответствующие известным значениям `city`, получат нули.у

Важно точно интерпретировать результат: нули не означают, что у объекта нет города. Они означают, что значение не совпало ни с одной категорией, изученной на train.

## Этап 6. Сделать результат наблюдаемым через `set_output`

### Учебная задача

Вернуть названия столбцов, чтобы ученики могли проверить смысл преобразования, не собирая `DataFrame` вручную.

```python
encoder = OneHotEncoder(
    handle_unknown="ignore",
    sparse_output=False
).set_output(transform="pandas")

train_encoded = encoder.fit_transform(train)
new_data_encoded = encoder.transform(new_data)
```

Теперь оба результата являются `DataFrame` с одинаковыми столбцами. Ученики проверяют:

```python
train_encoded.columns.equals(new_data_encoded.columns)
```

Ожидаемый результат:

```text
True
```

На этом этапе полезно связать техническую проверку с исходной задачей: пространство признаков осталось одинаковым, потому что его определили категории из train.

## Этап 7. Перейти к таблице со смешанными типами признаков

### Учебная задача

Развести два решения:

- категориальные столбцы нужно закодировать;
- числовые столбцы уже можно передать дальше.

Попытка применить `OneHotEncoder` ко всей таблице `students` закодирует также `age` и `score`. Синтаксически это возможно, но смысл признаков будет испорчен: каждое встреченное число станет отдельной категорией.

Ученики формулируют карту обработки:

```text
city, school_type -> OneHotEncoder
age, score        -> без изменений
```

Эта карта непосредственно подводит к `ColumnTransformer`.

## Этап 8. Собрать `ColumnTransformer`

### Учебная задача

Научить применять разные правила к разным группам столбцов.

```python
from sklearn.compose import ColumnTransformer

categorical_columns = ["city", "school_type"]

preprocessor = ColumnTransformer(
    transformers=[
        (
            "cat",
            OneHotEncoder(
                handle_unknown="ignore",
                sparse_output=False
            ),
            categorical_columns
        )
    ],
    remainder="passthrough",
    verbose_feature_names_out=False
).set_output(transform="pandas")

students_ready = preprocessor.fit_transform(students)
students_ready
```

Конструкция внутри `transformers` читается по трем позициям:

```text
имя шага -> преобразование -> список столбцов
```

Параметр `remainder="passthrough"` означает, что остальные столбцы проходят дальше без изменений. В данном примере это `age` и `score`.

### Проверка понимания

Ученик должен уметь по коду ответить:

1. Какие столбцы изучает `OneHotEncoder`?
2. Что произойдет с `age` и `score`?
3. Где после `fit` хранится список категорий?
4. Почему `ColumnTransformer` тоже нельзя заново обучать на test?

## Этап 9. Завершить цепочку с помощью `Pipeline`

### Учебная задача

Показать, что подготовка признаков является частью модели, а не отдельным ручным действием перед каждым прогнозом.

```python
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline

X_train = pd.DataFrame({
    "city": ["Москва", "Казань", "Омск", "Москва", "Казань", "Омск"],
    "school_type": [
        "лицей", "гимназия", "обычная",
        "лицей", "обычная", "гимназия"
    ],
    "age": [15, 16, 17, 16, 15, 17],
    "score": [78, 85, 64, 91, 73, 68]
})

y_train = [0, 1, 0, 1, 0, 0]

categorical_columns = ["city", "school_type"]

preprocessor = ColumnTransformer(
    transformers=[
        (
            "cat",
            OneHotEncoder(handle_unknown="ignore"),
            categorical_columns
        )
    ],
    remainder="passthrough"
)

model = Pipeline([
    ("preprocessor", preprocessor),
    ("classifier", LogisticRegression(max_iter=1000))
])

model.fit(X_train, y_train)
```

Новые данные передаются в исходном виде:

```python
X_new = pd.DataFrame({
    "city": ["Москва", "Тула"],
    "school_type": ["лицей", "обычная"],
    "age": [16, 15],
    "score": [88, 70]
})

predictions = model.predict(X_new)
```

Преподаватель разбирает два вызова как последовательность действий.

При `model.fit(X_train, y_train)`:

1. `preprocessor` изучает категории на `X_train`;
2. `preprocessor` преобразует `X_train`;
3. классификатор обучается на подготовленных признаках.

При `model.predict(X_new)`:

1. обученный `preprocessor` применяет сохраненное правило к `X_new`;
2. классификатор получает признаки той же структуры;
3. возвращаются предсказания.

Итоговая формулировка:

> `Pipeline` хранит порядок действий и обученные шаги, поэтому при прогнозе достаточно передать новые данные в исходном формате.

## Педагогические технологии и приемы

### 1. Проблемное обучение

Новая тема вводится через практическое противоречие: отдельное кодирование train и новых данных создает разные столбцы, хотя модель ожидает одну и ту же структуру. Ученики сначала фиксируют проблему, затем получают `OneHotEncoder` как средство ее решения.

### 2. Управляемое открытие

Преподаватель не сообщает сразу, почему произвольные коды не подходят. Ученики сравнивают две допустимые нумерации тех же городов и замечают, что числовой порядок зависит от выбора кодов. Вывод формулируется на основании наблюдения.

### 3. Обучение через контрастные примеры

На занятии последовательно сравниваются:

- числовой и номинальный признаки;
- произвольный числовой код и dummy-представление;
- ручные dummy-переменные и `pd.get_dummies`;
- отдельное кодирование двух таблиц и единое правило `fit`/`transform`;
- преобразование всей таблицы и выбор столбцов в `ColumnTransformer`;
- ручная последовательность действий и `Pipeline`.

Контраст должен отличаться одним существенным свойством. Это помогает ученику понять причину результата, а не запомнить внешний вид кода.

### 4. Цикл «предположение - наблюдение - объяснение»

Перед выполнением примера ученики прогнозируют результат:

1. Какие столбцы создаст `pd.get_dummies`?
2. Совпадут ли столбцы train и новых данных?
3. Что произойдет с неизвестной категорией при `handle_unknown="ignore"`?
4. Будет ли `Pipeline.predict` заново изучать категории?

После запуска кода прогноз сверяется с результатом, затем ученик объясняет наблюдение через изученное правило.

### 5. Пошаговое усложнение

В каждый момент добавляется только одна новая сущность:

```text
один категориальный столбец
-> два категориальных столбца
-> числовые и категориальные столбцы вместе
-> преобразователь и модель
```

Так рабочая память расходуется на новую идею, а не на одновременное чтение большого количества синтаксиса.

### 6. Разобранный пример с постепенным снятием опор

Сначала преподаватель полностью показывает создание dummy-переменных. Затем ученики дополняют вызов `OneHotEncoder`, самостоятельно указывают столбцы в `ColumnTransformer` и в конце собирают `Pipeline` по краткой схеме.

Последовательность заданий:

```text
готовый код -> код с пропусками -> сборка по схеме -> самостоятельное решение
```

### 7. Анализ ошибок

Ученикам предлагаются короткие фрагменты, в каждом из которых есть одна содержательная ошибка:

```python
encoder.fit_transform(X_train)
encoder.fit_transform(X_test)
```

```python
OneHotEncoder().fit_transform(students)
```

```python
pd.get_dummies(X_train)
pd.get_dummies(X_new)
```

Задача ученика - не только исправить код, но и назвать нарушенное правило:

- test не обучает преобразователь;
- разные типы столбцов требуют разных способов обработки;
- модель должна получать одну и ту же схему признаков.

### 8. Парное объяснение

Один ученик объясняет роль `fit`, второй - роль `transform`. Затем пара совместно разбирает, что происходит внутри `Pipeline` при `fit` и `predict`. Этот прием быстро выявляет смешение понятий «изучить правило» и «применить правило».

### 9. Формирующее оценивание

После каждого этапа задается один короткий вопрос, ответ на который определяет, можно ли двигаться дальше:

| После этапа | Диагностический вопрос |
|---|---|
| Произвольные коды | Какую лишнюю информацию получают модели? |
| Dummy вручную | Что означает один новый столбец? |
| `pd.get_dummies` | Откуда берется набор новых столбцов? |
| `OneHotEncoder` | Что сохраняется во время `fit`? |
| Неизвестная категория | Почему появляются нули во всех известных категориях? |
| `ColumnTransformer` | Какие столбцы меняются, а какие проходят дальше? |
| `Pipeline` | Что происходит с `X_new` перед прогнозом? |

Если ученик отвечает только названием метода, преподаватель просит описать изменение данных словами.

### 10. Практика воспроизведения

В конце занятия ученики без конспекта восстанавливают цепочку:

```text
OneHotEncoder
-> ColumnTransformer
-> Pipeline
-> fit
-> predict
```

После этого они подписывают, где категории изучаются, а где только используются. Такое задание проверяет не узнавание готового кода, а способность воспроизвести структуру решения.

## Типичные ошибки и методическая реакция

| Ошибка ученика | Возможная причина | Реакция преподавателя |
|---|---|---|
| Любой текстовый признак заменяет числами 0, 1, 2 | Считает любое число приемлемым входом модели | Предложить переставить коды и сравнить появившиеся порядок и расстояния |
| Называет все категории непорядковыми | Усвоил правило без границ применимости | Сопоставить `city` и уровни `низкий`, `средний`, `высокий` |
| Повторяет `fit_transform` на test | Не различает обучение и применение | Попросить подписать, какой объект меняет свое состояние |
| Кодирует всю таблицу | Ориентируется на тип команды, а не на смысл столбцов | Спросить, что будет означать dummy-признак `age_16` |
| Пытается вручную собирать `DataFrame` из результата энкодера | Не знает о табличном выводе sklearn | Использовать `.set_output(transform="pandas")` |
| Забывает неизвестные категории | Работает только с учебным train | Добавить в новые данные город, отсутствующий на train |
| Считает `Pipeline` способом сократить запись | Не видит связи между этапами | Проследить состояние шагов отдельно при `fit` и `predict` |
| Сразу применяет `drop="first"` | Воспроизводит найденный шаблон без понимания | На первом изучении оставить все категории; обсудить удаление позже в контексте линейных моделей |

## Типичные вопросы за рамками базового материала

### Всегда ли номинальные категории нужно кодировать через one-hot?

Нет. One-hot encoding является понятным и надежным базовым способом, но выбор зависит от модели, числа категорий и объема данных. Некоторые алгоритмы умеют работать с категориальными признаками напрямую. Для признаков с тысячами значений one-hot может создать слишком много столбцов, поэтому рассматривают группировку редких значений, hashing, частотное или target encoding.

Методический ответ должен начинаться не с названия энкодера, а с трех вопросов:

1. Есть ли у категорий содержательный порядок?
2. Сколько уникальных значений у признака?
3. Как выбранная модель работает с категориальными признаками?

### Если у признака две категории, обязательно ли создавать два столбца?

Нет, одну из двух категорий можно представить одним бинарным столбцом. Например, при значениях `да` и `нет` достаточно признака `is_yes`.

Однако при первом изучении полезно показывать полное one-hot представление: оно буквально отражает идею «одна категория - один столбец» и одинаково работает для двух и для нескольких категорий. Сокращение числа столбцов стоит вводить позже, когда понятна роль конкретной модели.

### Нужно ли всегда указывать `drop="first"`?

Нет. Этот параметр удаляет один one-hot столбец из каждой группы категорий. Он может быть полезен при разборе линейных моделей, где полный набор dummy-признаков вместе со свободным членом создает линейную зависимость.

Это не универсальное правило:

- деревьям решений удаление первого столбца обычно не требуется;
- регуляризованные линейные модели способны работать с полным набором признаков;
- после удаления одна из известных категорий представляется нулями во всей группе, что усложняет интерпретацию неизвестных значений.

Сначала ученики должны освоить полное кодирование, а затем обсуждать `drop` в связи со свойствами модели.

### У районов может быть порядок по центральности. Почему тогда используется one-hot?

Название района и центральность - не одно и то же.

Если столбец содержит названия `центр`, `спальный`, `пригород`, в них действительно можно предположить направление от центра к окраине. Но одной упорядоченности недостаточно, чтобы автоматически присвоить коды `0, 1, 2`. Такая запись дополнительно утверждает, что:

- влияние признака меняется в одном направлении;
- переходы между соседними уровнями сопоставимы;
- вся существенная разница районов описывается только центральностью.

Если эти предположения обоснованы задачей, можно отдельно создать порядковый или числовой признак, например `зона_центральности` или `расстояние_до_центра`. Само название района при этом можно оставить номинальным и закодировать через one-hot. Тогда модель получает две разные части информации: принадлежность к району и измеримую характеристику его расположения.

### Почему неизвестная категория превращается в нули, а не создает новый столбец?

После обучения размер и смысл входа модели должны оставаться фиксированными. Для новой категории у модели нет обученного коэффициента или другого изученного правила. Если при прогнозе добавить новый столбец, структура входных данных изменится.

Поэтому `handle_unknown="ignore"` сохраняет прежнюю схему и ставит нули во всей группе известных категорий. Это технически устойчивое поведение, но не способ выучить смысл новой категории. Если неизвестные значения начинают встречаться часто, нужно проверить изменение данных и при необходимости переобучить цепочку.

### Всегда ли в one-hot группе находится ровно одна единица?

Для известной категории и полного набора one-hot столбцов - да: в группе данного исходного признака будет одна единица.

Исключения:

- неизвестная категория при `handle_unknown="ignore"` дает нули во всей группе;
- удаленная через `drop` базовая категория тоже представляется нулями;
- пропуски обрабатываются в соответствии с выбранной стратегией;
- в итоговой таблице с несколькими категориальными признаками будет по одной активной категории в каждой группе, а не одна единица на всю строку.

### Что делать с пропущенными значениями?

Сначала нужно определить их смысл. Пропуск может означать:

- значение забыли указать;
- значение неизвестно;
- признак неприменим к объекту;
- ошибка возникла при сборе данных.

После этого выбирается единая стратегия. Например, пропуски можно заменить отдельной категорией `не указано` или добавить индикатор отсутствия значения. Важно, чтобы правило было включено в общую цепочку и одинаково применялось к train и новым данным.

### Если категории уже записаны числами, нужен ли энкодер?

Тип данных не определяет смысл признака. Коды района `101`, `205`, `310`, номера классов или почтовые индексы выглядят как числа, но могут оставаться номинальными категориями.

Проверочный вопрос:

> Имеют ли смысл арифметические операции и расстояния между этими значениями?

Если разность между кодами ничего не означает, признак нельзя считать числовым только из-за его типа `int`.

### Можно ли все-таки использовать `pd.get_dummies` для train и новых данных?

Технически можно, если вручную зафиксировать столбцы, полученные на train, а все следующие таблицы приводить к этой схеме. Например, отсутствующие столбцы потребуется добавить, лишние удалить и сохранить правильный порядок.

Проблема не в том, что `pd.get_dummies` запрещен, а в том, что управление состоянием остается на разработчике. `OneHotEncoder` и `Pipeline` делают это правило явной частью обученной модели и уменьшают вероятность рассогласования.

### Что происходит при очень большом числе категорий?

Число новых столбцов растет вместе с числом уникальных значений. Для идентификаторов пользователей, товаров, адресов или поисковых запросов one-hot представление может занимать слишком много памяти и плохо обобщаться на редких значениях.

Следующие шаги анализа:

1. Проверить число и частоты категорий.
2. Выделить редкие и неизвестные значения.
3. Рассмотреть объединение редких категорий.
4. Сравнить one-hot с другими способами кодирования.
5. Проверить качество решения на валидации, а не выбирать энкодер только по размеру таблицы.

### Зачем `OneHotEncoder` умеет возвращать разреженную матрицу?

В one-hot таблице большинство значений равно нулю. Разреженный формат хранит главным образом положения ненулевых элементов и поэтому экономит память при большом числе категорий.

Для маленького учебного примера используется плотный вывод и `DataFrame`, потому что его удобно рассматривать. Для больших данных нужно проверить, поддерживает ли выбранная модель разреженный формат, и не преобразовывать матрицу в плотную без необходимости.

### Нужно ли масштабировать one-hot признаки?

Обычно бинарные one-hot столбцы оставляют в диапазоне `0/1`. Числовые признаки могут потребовать масштабирования в зависимости от модели.

Особенно важно обсудить масштаб при использовании методов, основанных на расстояниях: крупные числовые значения могут сильнее влиять на расстояние, чем несколько бинарных признаков. Это уже отдельный этап проектирования `ColumnTransformer`: числовым и категориальным группам назначаются разные преобразования.

### Как one-hot encoding связан с кросс-валидацией?

В каждом разбиении кодировщик должен изучать категории только на обучающей части этого разбиения. Если сначала закодировать весь набор данных, а затем запустить кросс-валидацию, преобразователь получит информацию о категориях из валидационных частей.

Когда препроцессор включен в `Pipeline`, инструменты кросс-валидации обучают все шаги цепочки заново внутри каждого обучающего разбиения. Поэтому `Pipeline` важен не только для удобного `predict`, но и для корректной оценки модели.

### Что нужно сохранять после обучения: энкодер или модель?

Нужно сохранять всю обученную цепочку `Pipeline`. Отдельное сохранение модели без препроцессора опасно: для прогноза придется вручную воспроизвести список категорий, порядок столбцов и все остальные преобразования.

Полная цепочка фиксирует единый контракт:

```text
исходные столбцы -> подготовка признаков -> предсказание
```

## Вектор дальнейшего изучения One-Hot Encoding

После базового маршрута `dummy -> OneHotEncoder -> ColumnTransformer -> Pipeline` тему можно развивать по следующей линии:

```text
семантика категорий
-> устойчивость к пропускам и новым значениям
-> редкие категории и высокая кардинальность
-> разреженное представление
-> связь кодирования с типом модели
-> сравнение с альтернативными энкодерами
-> честная валидация всей цепочки
-> контроль данных после внедрения
```

### Направление 1. Семантика признака

Углубить различие между номинальными, порядковыми и числовыми признаками. Научиться отделять название объекта от его измеримых свойств: например, `район` от `расстояния_до_центра`, а `марка_автомобиля` от `мощности`.

Результат: способ кодирования выбирается по смыслу признака, а не по его внешнему типу.

### Направление 2. Неизвсдестные, редкие и пропущенные значения

Исследовать, как появляются новые категории, что считать редкой категорией и когда полезно объединять редкие значения в общую группу. Отдельно определить политику обработки пропусков.

Результат: преобразователь устойчив не только на подготовленном примере, но и на меняющихся данных.

### Направление 3. Размерность и формат хранения

Сравнить плотное и разреженное представление, оценить число создаваемых столбцов и расход памяти. Проследить, какие модели принимают разреженный вход.

Результат: ученик понимает вычислительную цену one-hot encoding.

### Направление 4. Связь с выбранной моделью

Разобрать:

- полный набор dummy-признаков и `drop` в линейных моделях;
- влияние one-hot признаков на расстояния в kNN;
- особенности разбиений в деревьях;
- модели с нативной поддержкой категориальных признаков.

Результат: параметры кодирования выбираются вместе с моделью, а не отдельно от нее.

### Направление 5. Альтернативные способы кодирования

Сравнить one-hot encoding с:

- `OrdinalEncoder` для обоснованно порядковых признаков;
- частотным кодированием;
- hashing;
- target encoding;
- нативной обработкой категорий отдельными алгоритмами.

Особое внимание требуется target encoding: использование целевой переменной создает риск утечки, поэтому такое кодирование должно обучаться только внутри train и внутри разбиений кросс-валидации.

Результат: ученик умеет выбирать способ кодирования по кардинальности, семантике и риску утечки.

### Направление 6. Валидация и подбор параметров

Включить `Pipeline` в кросс-валидацию и подбор гиперпараметров. Сравнивать не только модели, но и варианты обработки категорий в одной процедуре оценки.

Результат: качество каждого варианта измеряется на данных, которые не участвовали в настройке преобразования.

### Направление 7. Эксплуатация модели

Научиться сохранять полный `Pipeline`, проверять входную схему, считать долю неизвестных категорий и отслеживать изменение частот значений после внедрения.

Сигналы для проверки или переобучения:

- растет доля неизвестных категорий;
- появляются новые массовые значения;
- изменяется распределение известных категорий;
- входные столбцы меняют тип или смысл;
- качество модели снижается на свежих данных.

Результат: one-hot encoding рассматривается как часть жизненного цикла модели, а не одноразовая операция над учебной таблицей.

## Итоговое задание

Ученику выдается таблица:

```python
X_train = pd.DataFrame({
    "district": ["центр", "север", "юг", "центр"],
    "building_type": ["кирпичный", "панельный", "панельный", "монолитный"],
    "area": [48, 61, 54, 72]
})
```

Нужно:

1. Назвать тип каждого признака.
2. Указать, какие столбцы должны пройти one-hot encoding.
3. Собрать `ColumnTransformer`, оставив `area` без изменений.
4. Включить преобразователь в `Pipeline` со знакомой моделью.
5. Объяснить, что произойдет при прогнозе для ранее не встречавшегося района.

Оценивание проводится не только по работающему коду. Ученик должен связно объяснить:

- почему `district` и `building_type` не кодируются произвольными целыми числами;
- где энкодер изучает категории;
- почему новые данные не вызывают повторный `fit`;
- как `Pipeline` сохраняет одинаковую подготовку признаков.

## Критерии освоения темы

Тема освоена, если ученик:

- определяет, какие категориальные признаки являются номинальными;
- объясняет искусственный порядок и расстояния при произвольной числовой кодировке;
- читает и создает dummy-признаки;
- использует `pd.get_dummies` как средство быстрого преобразования таблицы;
- различает `fit`, `transform` и `fit_transform`;
- обрабатывает неизвестные категории через `handle_unknown="ignore"`;
- проверяет названия признаков в табличном результате;
- применяет `OneHotEncoder` только к выбранным столбцам;
- объясняет роль `remainder="passthrough"`;
- собирает `ColumnTransformer`;
- включает подготовку данных в `Pipeline`;
- прослеживает полный путь данных при `fit` и `predict`.

## Методический итог

Смысл темы можно зафиксировать одной последовательностью:

```text
dummy-переменные показывают идею
pd.get_dummies автоматизирует преобразование одной таблицы
OneHotEncoder изучает категории на train
ColumnTransformer назначает правила группам столбцов
Pipeline связывает подготовку признаков и модель
```

Главный результат занятия - ученик воспринимает one-hot encoding не как отдельную функцию, а как часть воспроизводимого процесса машинного обучения.
